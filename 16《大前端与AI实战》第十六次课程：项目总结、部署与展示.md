#### **第二部分：从本地到云端 - 容器化部署理论 (约60分钟)**

**教师讲解，作为理论知识普及：**

1.  **为什么需要部署？**
    *   “目前我们的应用只能在自己的电脑上运行(`localhost`)。部署就是把我们的应用放到一台24小时在线的、有公网IP地址的电脑（即云服务器）上，让全世界任何有网络的人都能访问。而使用容器化技术(Docker)部署，是现代最主流、最高效的方式。”

2.  **现代部署流程概览 (基于Docker)**
    ```mermaid
    sequenceDiagram
        participant D as 开发者
        participant G as Git仓库 (GitHub)
        participant R as Docker镜像仓库 (Docker Hub)
        participant S as 云服务器 (ECS/CVM)
        participant U as 最终用户

        D->>G: git push (推送代码)
        D->>D: 编写 Dockerfile
        D->>D: docker build (本地构建镜像)
        D->>R: docker push (推送镜像到仓库)
        D-->>S: SSH登录服务器
        S->>R: docker pull (从仓库拉取镜像)
        S->>S: docker run (运行容器化应用)
        U->>S: 访问域名 (e.g., feynman.com)
        S-->>U: 返回前端页面/API响应
    ```

3.  **核心工具介绍**
    *   **云服务器 (ECS/CVM):** “可以从阿里云、腾讯云等厂商租用。它就是一台远程的Linux电脑。”

    *   **`Docker` (容器化技术):** “这是现代软件开发和部署的基石。它能将我们的应用（包括代码、依赖、环境配置等）打包到一个独立的、轻量级的‘容器’里。”
        *   **解决什么问题？**
            1.  **环境一致性：** 彻底解决“在我电脑上能跑，在你服务器上就出问题”的世纪难题。容器在哪都一样运行。
            2.  **简化部署：** 无需在服务器上逐个安装Node.js, Nginx等。只需要安装Docker，然后运行`docker run`命令即可。
            3.  **隔离性与可移植性：** 每个容器都是一个隔离的环境，应用之间互不影响，可以轻松迁移。
        *   **后端 `Dockerfile` 示例:**
            ```dockerfile
            # 使用官方的 Node.js 18 镜像作为基础
            FROM node:18-alpine

            # 设置工作目录
            WORKDIR /app

            # 复制 package.json 和 package-lock.json
            COPY package*.json ./

            # 安装依赖 (在生产环境中建议使用 npm ci)
            RUN npm install

            # 复制所有项目文件
            COPY . .

            # 暴露后端服务端口
            EXPOSE 3000

            # 启动命令
            CMD [ "node", "index.js" ]
            ```
        *   **前端 `Dockerfile` (多阶段构建) 示例:**
            ```dockerfile
            # --- 构建阶段 --- 
            FROM node:18-alpine AS builder

            WORKDIR /app

            COPY package*.json ./
            RUN npm install
            COPY . .
            # 执行打包命令
            RUN npm run build

            # --- 生产阶段 ---
            FROM nginx:stable-alpine

            # 从构建阶段复制打包好的静态文件到 Nginx 的默认Web根目录
            COPY --from=builder /app/dist /usr/share/nginx/html

            # （可选）如果需要自定义Nginx配置，可以取消下面这行注释
            # COPY nginx.conf /etc/nginx/conf.d/default.conf

            # 暴露Nginx的默认端口
            EXPOSE 80

            # 启动Nginx服务
            CMD ["nginx", "-g", "daemon off;"]
            ```

    *   **`PM2` & `Nginx` (在非容器化或特定场景下使用):**
        *   “在我们了解了Docker之后，`PM2`和`Nginx`的角色就有了变化。在简单的Docker部署中，我们甚至可以不需要它们。”
        *   “`PM2`：Docker自带了重启策略（`docker run --restart=always`），可以替代PM2的进程守护功能。但在更复杂的场景或非容器化部署中，PM2依然是管理Node进程的好工具。”
        *   “`Nginx`：在前端的Dockerfile中，我们直接使用了Nginx镜像来托管静态文件。对于更复杂的场景，比如需要统一处理多个后端服务的API网关、配置SSL证书等，我们依然可以在服务器上独立运行一个Nginx容器作为所有流量的入口（反向代理）。”

    *   **教师说明：** “由于时间限制和操作复杂性，我们今天不进行实际部署操作，但大家必须理解**容器化部署**的流程和核心思想。这是从‘学生项目’到‘线上产品’的关键一步，也是现代软件工程师的必备技能。”

#### **第三部分：成果汇报与展示 (约90-120分钟)**

**学生主导，教师点评：**

1.  **汇报要求**
    *   **形式：** 以个人或小组为单位。
    *   **时长：** 每人/组 8-10分钟（5分钟演示 + 3-5分钟讲解/问答）。
    *   **内容：**
        1.  **项目演示 (Live Demo):** 动态展示你完成度最高、最有亮点的2-3个功能模块。例如：完整的RAG问答流程、炫酷的3D知识图谱、从录音到AI评价的全过程等。
        2.  **技术亮点：** 简要说明你在实现这个功能时，主要运用了哪些技术，解决了什么关键问题。
        3.  **心得体会：** 分享你在开发过程中的收获、遇到的困难或对未来的改进想法。

2.  **汇报顺序**
    *   可以按学号或自愿顺序进行。教师需控制好时间。

3.  **教师与同学点评**
    *   教师在每组汇报后进行简要点评，肯定优点，指出可改进之处。
    *   鼓励其他同学提问，营造积极的交流氛围。
